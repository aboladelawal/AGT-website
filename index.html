Our work

Built to prove the standard.

Two of our own platforms, engineered for one of the least forgiving environments there is. Financial messaging. Public architecture previews are open on GitHub.

Our platform Java 17 · Spring Boot 3

ISO 20022 Converter

A SWIFT MT to ISO 20022 MX message converter, both directions. It translates the formats banks have run for decades into the standard their counterparties now expect.

Architecture on GitHub
alpha-converter · operations console
  • Message coverage · MT103, MT202, MT940 and more
  • Mapped to MX · pacs.008, pacs.009, camt.053, camt.054
  • Header generation · Business Application Header, SAA envelope
  • Audited · full trail, reprocess, encrypted credentials
Our platform Java 17 · Spring Boot 3

SFTP — MQ Bridge

Integration middleware that moves files between SFTP and IBM MQ, both directions. For the common case where partners send files but internal systems read from queues.

Architecture on GitHub
alpha-bridge · operations console
  • Outbound · polls SFTP, publishes to a target queue
  • Inbound · listens on MQ, writes files back to SFTP
  • Routing · success, error, unsupported, with a replay archive
  • Resilient · retries, dead-letter queue, health checks
How we work

The part that doesn't change.

Whatever we build, the discipline underneath is the same. It is the reason the work holds up.

We would rather ship one system that holds under load than ten demos that impress in a meeting.
— Alpha Global engineering
01

Read the system first

We start in the code and the constraints as they actually are. No theorising before the facts are in.

02

Design for what fails

Retries, idempotency, reconciliation. We assume things break, and build so that nothing important breaks with them.

03

Keep config out of code

Endpoints, rules, rates all live outside the code. The same system adapts to a new case without a rebuild.

04

Hand over something owned

Documented, deployable, auditable. We leave a system your team can run without us.

How we engage

Questions worth asking first.

A few honest answers before you reach out.

How does a project start?

With a conversation. You tell us what you need, we scope it honestly, and we agree on what good looks like before any code is written. Most engagements begin with a short paid discovery so the estimate is real, not a guess.

Do you take small projects, or only large builds?

Both. We have built full platforms and we have fixed a single broken integration. What matters is whether the work is a fit for how we work, not its size. If it is not a fit, we will say so.

How does licensing work for your banking platforms?

The ISO 20022 Converter and SFTP-MQ Bridge are licensed for deployment in your environment. The public GitHub repositories are architecture previews. The full implementation, support, and configuration are provided under a commercial license. Talk to us about scope and tiers.

Do you work remotely, or only in Lagos?

We are based in Lagos and work with clients wherever they are. Engineering work is delivered remotely. For banking deployments that need on-site presence, we plan that into the engagement.

What does it cost?

It depends on the work, so we do not publish fixed prices. We quote per project after discovery, and for longer engagements we offer a monthly retainer. You will always know the number before we start.

Got something that needs building?

Tell us what you have in mind. We reply within two working days, usually sooner.
